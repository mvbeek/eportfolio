Introduction to Secure Systems Architecture


LECTURECAST SUMMARY

What is Secure Systems Architecture?

Secure systems architecture (SSA) brings together many disciplines including operating system
security, distributed systems security and communications security – amongst others.

Increasingly disciplines such as Network Security are focusing on areas that overlap within the groups
above, using tools like penetration tests and even attack graphs to calculate vulnerabilities.

Conrad et al (2012) define secure systems architecture and design as “describ(ing) fundamental logical
hardware, operating system, and software security components and how to use those components to
design, architect, and evaluate secure computer systems." They note that the ‘domain’ can be broken
down into three parts – the study of the hardware and software required to build a secure system,
the second being the creation of logical models required to keep systems secure, and the third part
covers “the evaluation models that quantify how secure the system really is” (Conrad et al, 2012).
These three approaches also define fundamental principles of the Security and Penetration testing
discipline.

Historically, that is in the pre-WWW days, most systems were stand-alone, and a secure system
test was performed on individual machines – not on network connected systems. Indeed, Windows NT4
only achieved its valued C2 security rating when it was NOT connected to a network (Russinovich,
1998).

Rushby’s Separability Model (1981), although old, makes some excellent points about security testing
of distributed systems and forms the basis of many papers as well as of formal methods around
security proofs of real systems. It posits the concept of a separation kernel which provides the same
control over connectivity that physically separated systems provide. It also separates the security policy
from the security mechanism. He concludes by recommending that the verification and testing of the
kernel is separated from the verification and testing of other secure, trusted processes.

This is particularly relevant to systems today as we move towards cloud-based systems where physical
(so-called ‘air-gapped’) isolation is increasingly difficult to achieve. This is amplified by the fact that the
underlying hardware of cloud platforms is based on multi-core processors and that some systems may
be implemented in their entirety on a single die.
